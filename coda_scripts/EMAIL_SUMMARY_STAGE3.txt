Subject: Stage 3 Updates - ROL Code Modifications for Config/Pedestal Handling

Hi Team,

Following Stages 1 and 2 (file-based component launching and automatic config generation), I've completed Stage 3 which moves configuration and pedestal handling from startup scripts into the ROL code itself.

## Summary of Changes

Stage 3 removes runtime pedestal generation and dynamic config file creation from the VME and VTP ROLs, replacing them with a cleaner approach that uses pre-generated configuration files.

## Modified Files

### 1. rol/vme_rol/fadc_vme.c

**What was removed:**
- Entire pedestal generation block (~187 lines)
- Runtime execution of fadc250peds command
- Dynamic generation of vme_hostname.cnf and vtp_hostname.cnf files
- parse_user_config() calls
- All fork/exec and child process monitoring code

**What was added:**
- Simple config file selection logic based on userconfig parameter
- If userconfig is defined: use it (preserves existing behavior)
- If userconfig NOT defined: use $CODA_CONFIG/vme_<hostname>.cnf

**Example default config path:**
```
$CODA_CONFIG/vme_test2.cnf
```

### 2. rol/vtp_rol/vtp_sro_1udp.c

**Firmware path updates (rocDownload):**
- Changed from hardcoded $CODA/src/vtp/firmware/ paths
- Now uses $CODA_FIRMWARE environment variable
- Fallback to $CODA/firmware if CODA_FIRMWARE not set
- Added file existence checks before loading firmware
- Enhanced error messages for missing firmware files

**Config file selection (rocPrestart):**
- If userconfig is defined: use it (preserves existing behavior)
- If userconfig NOT defined: use $CODA_CONFIG/vtp_<hostname>.cnf

**Example firmware paths:**
```
Primary:  $CODA_FIRMWARE/fe_vtp_z7_streamingv3_ejfat_v5.bin
Fallback: $CODA/firmware/fe_vtp_z7_streamingv3_ejfat_v5.bin
```

## Environment Variables

**CODA_CONFIG** (required)
- Base directory for configuration files
- Default VME config: $CODA_CONFIG/vme_<hostname>.cnf
- Default VTP config: $CODA_CONFIG/vtp_<hostname>.cnf

**CODA_FIRMWARE** (optional)
- Directory containing VTP firmware files
- If not set, falls back to $CODA/firmware

**CODA** (required if CODA_FIRMWARE not set)
- CODA installation root directory
- Used for firmware path fallback

## Workflow Changes

**Before Stage 3:**
1. startCoda generates config files (Stage 2)
2. ROL rocDownload() generates pedestals AGAIN (duplicate)
3. ROL rocDownload() generates config files AGAIN (duplicate)
4. ROL loads configuration

**After Stage 3:**
1. startCoda generates config files ONCE (Stage 2)
2. ROL rocDownload() checks userconfig parameter:
   - If defined: use specified file (backward compatible)
   - If NOT defined: use $CODA_CONFIG/vme_<hostname>.cnf
3. ROL loads configuration and proceeds

**Key improvements:**
- No duplicate pedestal generation
- No duplicate config file generation
- Faster run start times
- Simpler ROL code
- Config files can be inspected/validated before runs

## Backward Compatibility

**Existing behavior preserved:**
- Runs that define userconfig in CODA GUI work exactly as before
- Custom config file paths via userconfig still supported
- No changes required to existing run configurations

**New default behavior:**
- When userconfig is blank/undefined, ROLs use per-host default configs
- These defaults are generated once during initial setup via startCoda Stage 2
